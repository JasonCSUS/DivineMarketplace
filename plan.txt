DivineMarketplace plan
======================

Current implementation baseline (May 2026)
------------------------------------------
- Target platform: Minecraft/Paper 1.21.11 using Paper API 26.1.2.
- Runtime market state is SQLite-backed in `plugins/DivineMarketplace/data/market.db`.
- Legacy `.bin`, binary-cache, and owner-hash claim-file storage notes are obsolete.
- Player GUI/chat state is intentionally sharded by player UUID in memory so many players can use the marketplace at the same time. This is UI/session sharding, not separate claim persistence files.
- Item claims, money claims, active listings, sales history, admin audit history, market index, recommendations, price history, custom enchant metadata, custom item overrides, and small runtime state values all live in SQLite tables.
- Text YAML files still matter as bootstrap/editable definition sources: `config.yml`, `category_config.yml`, `categories/*.yml`, `custom/custom_items.yml`, `custom/custom_enchants.yml`, and `menu.yml`.

1. Startup and setup
--------------------
Purpose:
- create missing plugin folders and defaults on first startup
- preserve existing runtime database state
- generate readable admin helper files such as `permissions.txt`

Current behavior:
- `PluginFileInitializer` creates the live folder layout and copies missing bundled defaults
- `MainConfigLoader` parses `config.yml` into `MainConfig`
- `ConfigService` exposes cached config helpers
- `MenuVisualConfigLoader` loads and validates editable GUI layout/visuals from `menu.yml`
- `/market reload` reloads runtime config/data and refreshes the menu renderer without registering duplicate listeners

2. Runtime storage
------------------
Purpose:
- keep active market state durable without custom binary file maintenance

Current SQLite tables:
- `listings`
- `item_claims`
- `money_claims`
- `sales`
- `admin_sales`
- `admin_listings`
- `admin_claims`
- `market_index`
- `market_prices`
- `price_history`
- `runtime_state`
- `custom_enchants`
- `custom_item_overrides`

Rules:
- active listings are not size-purged as cache data; they are live market state
- SQLite storage tracking includes the main database file plus WAL/SHM sidecar files
- history table retention uses per-table logical payload size so shared SQLite storage still honors the configured history MB limits
- item claim cleanup uses the `item_claims` table payload size and abandoned claim age; it does not delete active/unexpired claims just because the total SQLite database is large
- retention cleanup checkpoints/truncates WAL and runs VACUUM after rows are deleted so old data can actually release disk pages
- `/market storage` shows current storage usage; `/market storage cleanup` runs the retention pass immediately
- write flows that touch multiple stores should use rollback helpers where practical

3. Item identity and definitions
--------------------------------
Purpose:
- resolve a raw `ItemStack` into one market-facing identity and behavior set

Resolved identity contains:
- `marketKey` for stable internal grouping
- `marketDisplayName` for player-facing labels
- `categoryId`
- optional `itemType` for custom items
- recommendation/history/training flags
- browse visibility
- review severity
- enchant map when relevant

Current custom-item direction:
- known config/definition files can seed custom item behavior
- runtime auto-discovered or admin-defined custom items write to SQLite-backed `market_index`
- category display names/icons/order come from `category_config.yml`; raw category ids remain internal command/storage tokens
- unknown/flagged custom item metadata should still be logged to readable files under `logs/` so server owners can inspect discoveries from the server computer
- newly discovered custom items default to `unsorted`, require admin review, and are excluded from recommendation training until confirmed

4. Listing flow
---------------
Purpose:
- let players sell a quantity from the item currently held in main hand

Current behavior:
- `/market list` or the GUI list button closes the menu and prompts in chat for `quantity unitPrice`
- chat prompt quantity tokens are exact integers, `stack`, and `all`
- `/market list <quantity> <unitPrice>` remains available as a direct command path with exact numeric quantity scope
- the server re-reads the live main-hand item before listing
- the requested amount clamps to the live main-hand stack, not matching stacks elsewhere in inventory
- the item is removed and verified before the listing is created or merged
- listing result objects report structured success/failure to commands and GUI prompts

5. Purchase flow
----------------
Purpose:
- safely transfer money to seller claims and purchased items to buyer claims

Current behavior:
- buyers cannot buy their own listings
- normal buyers see quantity controls and confirm purchase in item detail
- owners see inspect/cancel controls, not buy controls
- admins viewing someone else's listing see buyer controls plus admin cancel
- purchases require either a mergeable buyer claim or an open claim slot before money is withdrawn
- purchased items are routed to buyer item claims, not direct inventory delivery
- seller proceeds accumulate in money claims and are paid through Vault when claimed

6. Claim flow
-------------
Purpose:
- safely hold owed items/money until the player is ready to claim or relist

Current behavior:
- item claims and money claims are separate SQLite tables
- item claims merge similar items for the same owner when possible
- claim detail supports:
  - claim one safe stack-sized chunk
  - claim as much as safely fits in storage inventory contents
  - relist through chat input for `quantity unitPrice`
- claim actions return structured result objects so GUI/commands can show exact claimed amounts and remaining claim amount
- money claims pay accumulated seller earnings through Vault and return exact payout results
- relist-from-claim creates/merges a listing from the stored claim snapshot and decrements the claim only after successful listing persistence

7. History, auditing, and recommendations
-----------------------------------------
Purpose:
- separate admin audit history, player-facing sale history, and training data

Current behavior:
- admin history is mandatory for market actions and is stored in dedicated SQLite admin tables
- player-facing sale history is stored as `SaleRecord` entries in SQLite `sales`
- `InMemorySaleHistoryIndex` preloads recent sales for fast command/GUI reads
- normal sale history uses exact market-key matching
- enchanted-book sale history uses component matching:
  - single-enchant book history includes mixed-book sales containing that enchant+level
  - mixed-book history includes exact mixed sales plus matching single-enchant component sales
- mixed enchanted books remain player-visible in sale history but are excluded from recommendation training
- price history is stored as compact daily recommendation points in SQLite `price_history`
- price-history GUI month navigation only shows months that actually have data
- sale/price history GUI timestamps are rendered as simple server-local `yyyy-MM-dd HH:mm` values, not raw epoch milliseconds
- price history remains disabled for mixed-enchant book groups because they do not train recommendations

8. GUI and player input
-----------------------
Purpose:
- make `/market` the primary player experience while keeping text commands available

Current behavior:
- `/market` opens the main GUI
- `/market all`, `/market search <query>`, `/market list <quantity> <unitPrice>`, `/market pricecheck`, `/market history`, and `/market pricehistory` remain command paths
- GUI search, listing, and relisting use short-lived per-player chat prompts because inventory GUIs are poor text/price-entry surfaces
- prompt state is separate from menu sessions and keyed by player UUID
- menu sessions/actions are also keyed by player UUID, keeping simultaneous players isolated
- menu actions store only ids/tokens and always re-fetch live business data before mutation
- pagination loads `pageSize + 1` so Next only renders when another page exists
- price-history month navigation uses only months with stored history data
- `menu.yml` validation warns and falls back on invalid slots, duplicate slots, invalid materials, and invalid custom model data

9. Commands
-----------
Player-facing:
- `/market`
- `/market all`
- `/market <category>`
- `/market search [query]`
- `/market list [quantity unitPrice]`
- `/market pricecheck` or `/market pc`
- `/market claim`
- `/market claim earnings`
- `/market history <display name>`
- `/market pricehistory <display name> [menu]`

Admin-facing:
- admin commands remain behind focused handlers under `MarketAdminCommand`
- `/market storage` shows SQLite/table storage pressure and `/market storage cleanup` runs retention cleanup
- player suggestions should prefer display names
- admin suggestions may expose market keys where maintenance needs exact ids

10. Alpha cutoff goals
----------------------
Done/current for alpha:
- SQLite runtime storage path
- GUI browsing and detail views
- chat input for search/list/relist
- pricecheck command for held-item recommendations
- claim results with exact player feedback
- price-history GUI path from item detail and `/market pricehistory <item> menu`, with data-aware month navigation
- component-aware enchanted-book sale history

Known follow-up scans before hands-on server testing:
- inspect command output for any remaining internal market-key leaks to normal players
- review admin command UX and decide which flows should remain command-only for alpha
- manually verify `menu.yml` slot defaults after applying patches
- verify permissions in `plugin.yml` against `permissions.txt`, including `divinemarketplace.admin.storage`
- local compile was reported successful by the project owner; continue to use real server startup logs for runtime validation


11. Source comments and docs
----------------------------
Purpose:
- keep source comments useful for maintenance instead of preserving old planning notes

Current behavior:
- Java source files include a file-role comment near the package declaration describing the file's runtime responsibility
- source comments should describe current behavior; root docs should carry open planning work
- root planning stays in `plan.txt` and `PROJECT_STRUCTURE_TODO.md`; source-side planning maps should stay absent
- YAML defaults include short header comments explaining what server owners can safely edit
