## Alpha startup/install notes

- The plugin jar is built as a fat jar that bundles SQLite JDBC and HikariCP.
- Paper/Vault are still `compileOnly`/server-provided dependencies, but normal JVM libraries used by SQLite storage must be inside the plugin jar.
- If startup reports `NoClassDefFoundError: com/zaxxer/hikari/HikariConfig` or `org/sqlite/JDBC`, rebuild with the current `build.gradle.kts` and deploy the newly generated jar from `build/libs`.

DivineMarketplace plan
======================

Current implementation baseline (May 2026)
------------------------------------------
- Target platform: Minecraft/Paper 1.21.11 using Paper API 26.1.2.
- Runtime market state is SQLite-backed in `plugins/DivineMarketplace/data/market.db`.
- Legacy `.bin`, binary-cache, and owner-hash claim-file storage notes are obsolete.
- Player GUI/chat state is intentionally sharded by player UUID in memory so many players can use the marketplace at the same time. This is UI/session sharding, not separate claim persistence files.
- Item claims, money claims, active listings, sales history, admin audit history, market index, recommendations, price history, custom enchant metadata, custom item overrides, and small runtime state values all live in SQLite tables.
- Text config files still matter as bootstrap/editable definition sources: `config.yml`, `category_config.yml`, `categories.csv`, `custom/custom_items.yml`, `custom/custom_enchants.yml`, and `menu.yml`.
- Bundled `defaults/categories/*.yml` files are internal seed data. Startup converts them into one live `categories.csv` file instead of copying a folder full of YAML files.



Current top priority: async-safe market runtime
-------------------------------

The next refactor priority is not another file move. The plugin must become safe under real multiplayer marketplace pressure. The target is:

- Keep all Minecraft/Paper-facing work on the main server thread: inventory clicks, chest GUI rendering/opening, player inventory reads/writes, item delivery, messages, and Vault/economy calls unless proven safe.
- Move plain plugin data work async where possible: SQL load/write, sorting/filtering snapshots, search preparation, analytics, price recalculation, and history/export work.
- Load SQLite storage into memory during startup, then treat memory as runtime truth. Normal menu/command reads should not query SQL.
- Keep WAL enabled, but still route all writes through one controlled SQLite write pipeline because SQLite has one writer.
- Batch writes by market action. A purchase/cancel/claim/relist should enqueue one transaction-sized batch instead of letting several stores write separately.
- Purge or quarantine old direct append/save/delete methods after queued equivalents are stable, so future code cannot accidentally bypass the writer queue.
- Flush critical writes on a timer/threshold and during shutdown. Prefer roughly 30-60 seconds for normal batching, with sooner threshold flushes for purchases/claims. Five minutes is acceptable only for low-risk analytics, not economy-critical state.
- Use immutable snapshots for async reading/planning. Async workers should calculate deltas, then the runtime should validate and apply those deltas to current authoritative memory. Do not mutate a copied snapshot and blindly replace current memory.
- Add per-player and per-object action locks so spam clicks cannot create duplicate purchases, duplicate listing cancels, duplicate item claims, or duplicate money payouts.
- Async as many menu preparation tasks as possible, but do not async the Bukkit inventory GUI itself. Expensive page/search preparation can happen from snapshots; final ItemStack rendering/opening stays sync.
- Treat GUI performance as a top-level alpha hardening item: the current renderer rebuilds inventories/page data from raw Java methods on each open/page change, so the next GUI refactor needs templates, cached plain page models, data-version invalidation, and async page preparation for heavy screens.
- Short-term future work: ~~merge admin history and player market/sale history into one canonical market event stream~~ (DONE — market_events canonical store implemented).
- Cleanup direction: move back toward time-based retention/deletion instead of storage-size-based deletion to reduce repeated size calculations and make admin behavior easier to reason about.
- Purge/delete/maintenance operations must be mutually exclusive with the normal write-behind worker: drain or pause normal queued writes first, run the maintenance transaction, then resume the writer.
- Reload should become targeted and async: config/menu/prices can reload by command, but full SQL-backed runtime reload should not run synchronously on the server thread. Reload paths should keep WAL enabled and use the same storage worker/maintenance gate.

See `ASYNC_MARKET_RUNTIME_PLAN.md` for the detailed thread/storage/snapshot plan.

Major architecture redo target
------------------------------
This is the next large refactor after the current alpha behavior is stable. The goal is not to throw away working marketplace behavior; the goal is to move the existing services, registries, stores, GUI code, command handlers, and helpers into a clearer architecture that is easier to read, extend, and optimize.

Guiding principles:
- Rebuild the project layout around an explicit Registry / Service / Store framework.
- Keep the main plugin class small. It should own Paper lifecycle handoff only: create the runtime, enable it, disable it, and expose only the minimal plugin-level accessors Paper needs.
- Move startup wiring into module/bootstrap classes so `DivineMarketplace.java` is not a long list of construction and dependency glue.
- Reduce duplicated player/admin data. Player views, admin logs, Discord webhooks, and analytics should be projections of the same canonical market data/events whenever possible.
- Make runtime reads memory-first. Minecraft servers should spend RAM to avoid repeated CPU-heavy scans and repeated SQL IO during normal gameplay.
- Make SQL durability/recovery/reporting, not the hot gameplay path.
- Batch writes aggressively. Runtime mutations should update memory immediately, enqueue a persistence operation, and flush/coalesce writes in the background or at shutdown.
- Avoid frequent destructive storage cleanup during normal server runtime. Deletion/compaction should happen at startup and through explicit admin commands, not on a short recurring schedule.
- Destructive cleanup and reload maintenance must coordinate with the write-behind worker so normal writes and purge/delete transactions never run against SQLite at the same time.
- Document dependencies between files/modules so a new maintainer can follow which layer owns what.

Target module layout:
- `bootstrap` / `runtime`: plugin lifecycle, module creation, dependency wiring, enable/disable ordering.
- `storage`: SQL connection management, schema migrations, write-behind queue, storage backend abstractions, SQLite/MySQL implementations.
- `store`: memory-first stores and indexes. Stores load SQL rows at startup, treat memory as runtime truth, and enqueue persistence writes.
- `registry`: definition/lookup data such as categories, item identities, custom items, custom enchants, permissions, menu definitions, and market key metadata.
- `service`: business behavior such as listing, buying, claiming, relisting, recommendations, history, storage maintenance, admin review, and event publication.
- `event` or `market_event`: canonical marketplace events. Admin history, player history, Discord webhooks, and audit exports should be generated from these events instead of separate duplicated data paths.
- `gui`: a strict inventory GUI framework plus marketplace screens. Avoid freehand menu construction in individual render methods.
- `command`: command entry points that delegate to services/GUI/prompt flows and do not contain business logic.
- `prompt`: chat input sessions for search/list/relist/price entry.
- `scheduler`: named scheduled tasks and startup/shutdown flush hooks.
- `util`: shared MiniMessage/message formatting, money formatting, time formatting, item formatting, pagination helpers, permission helpers, scheduler helpers, config validation, and small collection/index helpers.

Registry / Service / Store responsibilities:
- Registries answer definition questions: what category exists, what item identity means, what custom enchant is known, what GUI material/icon should be used.
- Services perform marketplace behavior and enforce rules. Services may use registries and stores, but should not know SQL details.
- Stores own cached mutable state and persistence queues. Stores expose indexed reads and mutation methods; they should not contain player-facing messages or GUI behavior.
- Commands, GUI, and chat prompts are adapters. They gather player input, call services, and render service results.

Memory-first storage target:
- On enable/reload, stores load all required active rows into memory.
- Runtime reads use in-memory maps/indexes only.
- Runtime writes update memory synchronously, then enqueue write-behind SQL operations.
- The write-behind queue coalesces repeated updates to the same logical record where safe.
- The queue flushes on a timer, on operation thresholds, before reload/maintenance, and during plugin disable.
- Shutdown flush must block long enough to make pending writes durable or log a clear failure.
- SQL reads during gameplay should be exceptions, not the default pattern.
- Listing stores should maintain indexes such as by id, seller, category, market key, and searchable display tokens.
- Claim stores should maintain indexes by owner and claim id.
- History stores should maintain recent in-memory indexes for GUI/command reads and write older durable rows to SQL.

Storage cleanup target:
- Current code has scheduled SQLite retention cleanup. That is acceptable for the current alpha implementation, but it conflicts with the long-term target.
- Refactor target: destructive cleanup/compaction should run on startup and through `/market storage cleanup`; it should not run frequently while players are using the market.
- Runtime storage pressure can still be monitored periodically, but monitoring should not delete rows or VACUUM while the server is active unless an admin explicitly requests it.
- Purge/delete/compaction must be mutually exclusive with normal queued writes. The maintenance path should stop accepting new write batches, flush or checkpoint pending batches, run the maintenance transaction, then reopen normal queued writes.
- Reload/maintenance commands should keep SQLite WAL enabled and should not silently fall back to a blocking main-thread SQL path.
- SQLite and MySQL backends should share the same store/service contract. Backend-specific cleanup should live under storage backend implementations.

Canonical market event direction:
- Create one event model for marketplace changes: listing created, listing merged, listing cancelled, item purchased, item claim created, item claim redeemed, claim relisted, money claim created, money paid, recommendation changed, admin override applied, custom item flagged, etc.
- Player sale history, admin audit history, analytics, Discord webhooks, and export tools should read from this shared event stream.
- Keep separate projections/tables only when they serve a clear performance or retention purpose.
- Admin history should not duplicate sale history as a second unrelated write path when both describe the same purchase.
- Discord webhooks should subscribe to event projections rather than being called manually from many unrelated services.

Public-server expansion goals:
- Optional MySQL backend for larger public servers.
- Config-driven storage backend: `sqlite` by default, `mysql` optional later.
- HikariCP is already used for SQLite and should remain for MySQL.
- Services, commands, and GUI should not care which backend is active.
- Optional Discord webhooks for listing created, purchase, rare item sale, admin actions, custom item flagged, daily market summary, and storage errors.
- Webhook delivery should be queued/batched and rate-limited.
- Add readable local text logs for admin review and flagged custom item discovery even when SQL remains the source of truth.

Current alpha checklist
-----------------------
Keep these as the practical near-term tasks before another large feature wave.

Async/storage hardening:
- Finish direct-write compliance. All remaining normal runtime writes should go through the queued write-behind path unless they are startup-load, explicit admin maintenance, or shutdown-flush operations.
- Purge old direct append/save/delete APIs after queued replacements are confirmed. Methods like old `append(...)`, `saveOrReplace(...)`, `delete(...)`, and direct helper writes should become private, startup/admin-only, or be removed so future menu/service code cannot bypass the queue.
- Convert price recalculation completion to snapshot -> delta -> memory update -> queued SQL batch. It should not require reload.
- Convert runtime-state writes such as last global recalculation date to queued SQL batch.
- Decide whether admin/custom metadata writes should use the same queue or remain explicit admin-only blocking operations. Prefer the queue unless blocking is required for command feedback.
- Keep startup SQL reads async and guarded by runtime state.
- Make `/market reload` targeted and async: reload config/menu/prices through specific command paths; do not synchronously reload all SQL-backed runtime data on the server thread.
- Reload should coordinate with SQLite WAL and the same worker/maintenance gate used by writes. Do not create a second unsynchronized SQLite path for reload reads/writes.
- Add a storage maintenance gate so purge/delete/VACUUM/checkpoint operations are mutually exclusive with the main write-behind worker.
- Add storage recovery, not a degraded-mode dashboard: if a queued SQLite flush fails, pause marketplace actions, keep memory/queue intact, reconnect SQLite/Hikari, retry the same queued flush, and mark runtime FAILED only after recovery retries fail.
- Add better shutdown-flush logging: number of pending batches/mutations, success/failure, and timeout behavior.

GUI performance hardening:
- Add a GUI template/model layer before adding more screens.
- Cache static menu pieces: filler panes, border buttons, back/next/previous buttons, disabled buttons, static labels, and title formats.
- Cache plain page models separately from Bukkit `Inventory` objects.
- Add data-version invalidation so cached pages expire when listings, claims, prices, or categories change.
- Use snapshots for async preparation of expensive pages/search/history results.
- Keep Bukkit `Inventory`, `Player`, and final `ItemStack` rendering/opening on the main thread.
- Convert one heavy screen first, likely listing browser or search results, then port sale history, price history, and category/group screens.
